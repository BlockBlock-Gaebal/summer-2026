## 0. 한 줄 요약

**의지력에 돈을 거는 탈중앙 습관 챌린지 프로토콜.** 참여자들이 목표(예: 미라클 모닝)를 정하고 돈을 예치, 실패하면 예치금을 잃고 성공자들이 나눠 갖는다. 핵심 차별점은 **시간가중 차등 몰수(day1 탈락 ≠ day50 탈락)등 일반적으로는 넣기 힘든 복잡한 돈 정산 시스템** 를 스마트 컨트랙트로 신뢰 불필요하게 자동 집행하는 것. <--- 그렇지 않으면 블록체인이 필요 없기에

### 팀의 현재 실력 수준 (AI 에이전트가 반드시 인지할 것)
- Git: pull/add/commit/push/브랜치/PR 개념 이해 + 실습 1회 완료 (자기소개 md 올리고 merge까지)
- **⚡ Move: 전원 미학습.** (진모의 Solidity 기초 학습분은 "컨트랙트적 사고"로만 이전됨 — 문법/멘탈모델은 새로 배워야 함)
- FE/BE: 환경 세팅만 완료, 실제 화면/API 구현은 아직

### ⚡ Sui/Move 핵심 차이 (Solidity 대비 — 팀 전원 숙지)
- **돈 = 객체(Object)**: Solidity는 잔고가 mapping 속 숫자지만, Move는 `Coin`이 실물 같은 리소스 객체. 복사 불가·실수로 소멸 불가(linear type). "돈이 새는" 버그를 언어가 원천 봉쇄 → 우리 같은 돈 중심 로직(Vault 예치→분할→분배)과 궁합 좋음
- 컨트랙트가 아니라 **패키지/모듈**이라 부름. Challenge는 **shared object**로 만들어 여러 참여자가 접근
- 시간: `block.timestamp` 대신 **`Clock` 시스템 객체**를 함수 인자로 받음 (UTC 밀리초 — KST 보정 문제는 동일하게 존재)
- 소수점 없음은 동일: u64 정수 연산 + 스케일링 (Solidity의 1e18 관행 대신 자릿수는 설계 시 결정)
- **주의**: Sui Move는 자료·AI 학습데이터가 Solidity보다 적음 → Claude Code가 옛 문법을 섞을 수 있으니 **공식 문서(docs.sui.io) 대조 검증 필수**
- **⚡ `Table`은 키 순회 불가** — 참가자 순회가 필요하면 `vector<address>`를 병행 유지해야 함 (상세는 정산_프로토타입/02_스펙)

## 2. 프로젝트 아이디어 (팀 확정안)

### 2-1. 기본형 (코어 메커니즘)

예시: **미라클 모닝 챌린지** <--- 미라클 모닝을 예시로 드는 이유는 그게 제일 쉽기 때문, 가령 매일 헬스하기 이런건 우리가 검증할 수 없음

1. 참여자들이 규칙을 정함 — "매일 아침 6시에 기상 보고"
2. 각자 돈을 예치 (예: 인당 10만 원) → 스마트 컨트랙트(Vault)에 잠김
3. 챌린지 시작, 매일 체크인 (온체인 기록)
4. 실패하면 본인 예치금이 Vault(Pool)에 몰수됨
5. 챌린지 종료 후, 몰수분이 **성공자들에게 분배**됨

```
[참여자 A,B,C,D] --예치--> [Vault (스마트 컨트랙트)]
        |
        |  매일 체크인 (온체인 기록)
        v
   실패자 예치금 몰수 --> 종료 시 성공자에게 분배
```
그런데 여기까지만 하면 web2랑 다를게 없기에 web3의 장점을 만들기 위해서는 더욱 응용해야함
### 2-2. 확장 아이디어 (스마트 컨트랙트라서 가능한 것들)

| 확장 | 설명 | 비슷한 DeFi 개념 |
|---|---|---|
| 불특정 다수 챌린지 | 아는 사람뿐 아니라 인터넷의 모르는 사람과도 가능 | Trustless 에스크로 |
| 가변 예치금 | 사람마다 다른 금액, 지분 비례 정산 | Uniswap V2 LP 토큰 지분 계산 |
| 실시간 스트리밍 보상 | 종료 후 일괄이 아니라 실시간 이자처럼 지급 | Superfluid/Sablier 머니 스트리밍 |
| **시간가중 차등 몰수** ★ | day1 탈락자와 day50 탈락자를 다르게 정산 | 시간가중 스테이킹 보상 |
| 중도 참여/중도 정산 | 유동적 입출금 | Aave/Compound share 기반 회계 |

구체적 수식은 금융공학적으로 유도 완료 — 5절 및 `정산_프로토타입/01_설계문서` 참조.
★ = 경쟁 리서치 결과 **아무도 명시적으로 구현하지 않은 white space**로 확인됨. 우리의 핵심 차별점.

### 2-3. "그냥 내기 어플 아님?" — 블록체인 당위성

- **Trustless**: 모르는 사람과 돈 걸기 = "쟤가 들고 튀면?" → 컨트랙트가 에스크로, 아무도 손 못 댐
- **자동 집행**: 운영자의 "걔는 사정 있으니 봐주자" 개입 원천 차단. 조건 미충족 = 무조건 집행
- **실증 근거 (킬러 탄약)**: 미국 GymPact(Pact) 사례 — 회사가 예치금 쥐고 판정하다 성공자에게도 과금, FTC 제재($948,788 소비자 환불 명령) 후 폐업. **중앙화 커스터디+판정의 신뢰 붕괴가 실제로 일어남** → 우리는 그 판정/집행을 코드에 위임
- **유연한 금융 로직**: 차등 몰수, 지분 비례, 중도 정산 같은 복잡한 정산일수록 중앙 운영자는 신뢰 문제가 커지는데, DeFi 프리미티브로는 자연스럽게 구현됨

### 2-4. 이론적 근거 (발표용)

- **커밋먼트 디바이스(Commitment Device)**: 행동경제학 개념, 미래의 나를 구속하는 자기계약
- **손실 회피(Loss Aversion)**: 잃는 고통 > 얻는 기쁨 (약 2배) → "실패 시 몰수"가 "성공 시 보상"보다 행동 변화 효과 큼
- **준쌍곡 할인(β-δ, Laibson 1997)**: 현재편향을 자각하는 유저가 자발적으로 구속 장치에 지불 — 우리 유저의 존재 근거. 상세 이론은 `정산_프로토타입/01_설계문서`

## 3. 검토에서 발견된 허점과 대응 (셀프 레드팀 결과)

### 🔴 허점 1: 봇 취약점
- 지갑 서명 체크인은 cron job으로 완전 자동화 가능 → "깨어있을 필요조차 없음"
- 오픈 챌린지 시 봇 파밍 공격 가능 (에어드랍 파밍과 동일 구조)
- **대응**: MVP는 "신뢰 기반 소그룹용"으로 스코프 정면 명시. 오픈 챌린지는 인증 강화(사진+AI, 웨어러블) 선행 조건으로 로드맵 처리. 지분 비례 분배 자체가 시빌 분할의 이득을 0으로 만드는 부수 효과도 있음 (설계문서 §3)

### 🔴 허점 2: 인센티브 꼬임
- 몰수금 재분배 구조 = 내 보상이 타인의 실패에 의존 → "같이 갓생 살자"인데 구조적으로 팀원 실패를 응원하게 됨
- 전원 성공 시 보상 0원
- StickK 비유는 부정확 (StickK는 몰수금이 자선단체로 감, 재분배 아님)
- **대응 옵션**: PoolTogether식 No-Loss 구조 — 예치금을 렌딩에 넣어 이자 발생, 성공자는 원금+이자, 실패자는 원금 반환+이자만 포기. 단 GoodGhosting/HaloFi가 "저축 습관"에 이미 적용했으므로 "세계 최초" 주장 금지, "비금융 실생활 습관 + 차등 몰수 결합" 각도로만   <------잘 모르겠음 실패시 돈 몰수는 무조건 있어야 하긴 할 듯
- **설계 반영**: "전원 완주 시 기대수익 0"은 오히려 투기적 진입을 차단하는 의도된 비대칭으로 재해석 — "도박이 아니다" 방어의 수학적 실체 (설계문서 §1의 IR 논의)

### 🟡 허점 3: "성공"의 정의 부재
- 30일 중 29일 성공 + 1일 실패자는? 1회 실패 = 즉시 전액 몰수는 너무 가혹 (리텐션 붕괴)
- **옵션**: (a) 임계값 방식(85% 이상 = 성공, 챌린저스 방식) (b) grace day (c) 실패 1회당 부분 몰수
- **미결정. 회의 최우선 안건** — 컨트랙트 상태 설계가 이것에 따라 달라짐. **커브 확정의 선행 조건이기도 함** (탈락일 d의 정의가 바뀌므로)

### 🟡 허점 4: 기술 함정 4종 (⚡ Sui 기준으로 갱신)
1. **정산 루프 가스 문제**: 성공자 전원에게 한 트랜잭션으로 분배하는 설계는 위험 → **pull 패턴** (각자 `claim` 호출해 찾아가기) — Move에서도 동일한 원칙
2. **타임존**: Sui의 `Clock` 객체는 UTC 밀리초. 한국 아침 6시 = UTC 21시. day 경계 계산 시 KST 보정 필수 (Solidity 때와 동일한 함정)
3. **네트워크 혼잡/지연**: 윈도우 마감 직전 제출이 지연되면 억울한 실패 → 시간 윈도우 여유 마진 (Sui는 처리속도 빨라 리스크 낮은 편이나 마진은 유지)
4. **데모 문제**: 30일 챌린지를 발표 5분에 시연 불가 → 기간/윈도우를 생성 파라미터로 분리, "1 day = 30초" 데모 모드 지원

### 🟢 허점 5: UX/역할분담
- 새벽 6시에 지갑 서명 = UX 지옥 → 로드맵에 세션 키/zkLogin(Sui의 소셜 로그인) 언급 — ⚡ Sui는 zkLogin으로 구글 계정 로그인 지원, 오히려 EVM보다 UX 개선 여지 큼
- 컨트랙트 파트만 신규 학습량 압도적 → "컨트랙트 2명 + FE/BE 겸임 1명" 또는 전원 스터디 후 분담 검토

## 4. 경쟁/유사 프로젝트 분석 (리서치 완료)

### 4-1. 가장 중요한 비교 대상

| 프로젝트 | 유형 | 유사도 | 핵심 차이 |
|---|---|---|---|
| **챌린저스** (한국, 웹2) | 습관 챌린지 앱 | **90%+** | 85% 달성 시 환급, 100% 달성자가 몰수금 분배받음. 누적 거래액 3000억+. **중앙화**: 회사가 예치금 커스터디/판정/수수료 통제 |
| **GymPact/Pact** (미국, 폐업) | 운동 내기 앱 | 높음 | 실패자 벌금 → 성공자 분배. **FTC 제재 후 폐업** — 우리 당위성 서사의 핵심 증거 |
| **StickK** | 커밋먼트 계약 | 중간 | 몰수금이 자선단체/anti-charity로 감 (**재분배 아님**) |
| **PumpRoyale** (ETHGlobal SF 2024 수상) | 온체인 피트니스 | 높음 | flat 몰수 + 랜덤 분배. 시간가중 없음 |
| **Chained Habits, Wake Me Up!, Proof of Workout** (ETHGlobal) | 온체인 습관 | 높음 | 전부 flat 몰수. 해커톤 PoC 수준 |
| **GoodGhosting/HaloFi** | No-Loss 저축 | 중간 | PoolTogether식 무손실을 **저축 습관에만** 적용 (라이브). 비금융 습관은 미점유 |
| **Social Kontracts** (ETHGlobal) | 온체인 습관 | 중간 | Superfluid 스트리밍 보상 선례 존재 (PoC 1건) → 스트리밍 "세계 최초" 주장 금지 |

## 5. 정산 커브 설계 (제안안 — 상세는 정산_프로토타입/01_설계문서)

커브를 "방향 선택" 문제로 다루지 않는다. 대신 인센티브 조건 — **하루 더 버티는 것의 가치(생존 임금)가 노력 비용을 전 구간에서 이겨야 한다** — 을 정의하고, 그 조건을 만족하는 함수족을 유도했다. 행동을 결정하는 것은 몰수액의 수준이 아니라 기울기다.

- **환급 커브**: `r(d) = α·(d/D) + (1−α)·(d/D)²` — 선형·이차의 볼록결합. 늦게 탈락할수록 더 챙기고(공정성), 하루 더 버티는 가치는 뒤로 갈수록 커짐(인센티브). α는 유일한 튜닝 파라미터 — 권장 0.3, 탐색 [0.2, 0.4]
- **분배 스케줄**: 몰수액을 즉시 분배하지 않고 잔여 기간에 걸쳐 균등 방출하는 **스트리밍(베스팅)**. 배당 흐름이 단조증가가 되어 후반 이탈의 기회비용에 확정적 하한이 생김. "재분배는 버틴 기간 비례"라는 성질은 별도 규칙 없이 스케줄 자체에서 창발
- **증가형 커브(sunk cost 압박형)는 명제로 기각됨**: 합리적 저신념 유저의 즉시 손절을 유도하고, 편향 착취라 발표장에서 방어 불가
- 유도·명제·증명·시뮬레이션은 `정산_프로토타입/01_설계문서/01_CURVE_DESIGN_ver1-3.md`, 중도참여(ver4) 설계는 `02_VER4_DESIGN.md`, 구현 스펙은 `02_스펙/PROTO_SPEC.md`

## 11. 설계 방향 (팀 확정 전 제안 포함)

### 11-1. 챌린지 스코프: "미라클 모닝 앱"이 아니라 "시간 기반 체크인 챌린지"
- 챌린지 종류는 하나(지정 시간 윈도우 내 체크인)로 좁히되, **시간은 유저가 자유 설정**
- 헬스/공부 등은 검증 수단이 달라 이번 스코프에서 제외

### 11-2. 커스텀 챌린지 방 구조
- 유저가 방을 직접 팜: "아침 7시 미라클 모닝 방", "밤 12시 방" 등
- 친구 초대(비공개=화이트리스트) 또는 공개방(외부인 참여 가능)
- ⚡ 구현: Sui에서는 **Challenge를 shared object로 생성** — 방 하나 = shared object 하나. 방 파기 = `create_challenge(체크인시간, 윈도우, 기간, 예치금, 공개여부)` 호출 (데모 때 즉석 방 생성 시연 가능)

### 11-3. 정산 수식의 위치: 무조건 스마트 컨트랙트
- 돈을 만지는 로직(몰수 커브, 분배 수식)은 전부 온체인 — 이게 우리 당위성 그 자체 ("규칙이 코드로 박제, 아무도 못 바꿈")
- ⚡ 제약 대응 완료: Move도 소수점 없음(u64/u128 정수 + 스케일링), 지수/로그 비쌈 → **볼록결합 커브는 2차 다항식이라 곱셈만으로 정확 계산됨** (지수 근사 불필요 — 커브 선택 시 IMPL 제약을 반영한 결과)
- 유저 커스터마이징 = 수식 직접 입력이 아니라, **config 선택 + 파라미터(α 등) 조정** 방식 (ver5)

### 11-4. 하이브리드 인증 아키텍처 ("돈은 온체인, 인증은 오프체인")
- **온체인**: 예치, 정산 수식, claim — 신뢰 불필요 영역
- **오프체인(BE)**: 체크인 인증(채팅/사진 등 소셜 상호작용 기반), 결과 집계 → BE가 결과를 컨트랙트에 제출 (오라클 역할)
- 이득: ① 유저가 매일 트랜잭션 안 날려도 됨 (가스비 + 새벽 지갑 서명 UX 해결) ② 소셜 인증(같이 채팅, 상호 확인)은 봇 공격도 완화 ③ 소셜 플랫폼 성격 부여
- **인정할 타협**: 인증 판정 자체는 중앙화됨 (거의 모든 실전 프로젝트의 표준 타협 — Proof of Workout도 Strava 의존). 발표 논리: "돈의 커스터디와 정산 규칙은 온체인이라 저희도 못 건드립니다. 인증 판정은 현재 오프체인이며, 향후 참여자 다수 서명/낙관적 이의제기로 탈중앙화 가능합니다"
- 스코프 주의: 영상통화(WebRTC)는 4주 내 무리 → MVP는 **채팅 + 사진 체크인 + 참여자 상호 확인**, 영상통화는 로드맵
- **⚠️ 검증 시스템 상세 설계(커뮤니티 투표의 담합 방지 등)는 논의했으나 후순위로 결정 (2026-07-16)** — 핵심 결론만 기록: 같은 방 상호검증은 게임이론상 담합이 우월전략이라 기각. 대안 후보 = 최소인원+익명화, 무작위 외부 심사, 심사 스테이킹(이더리움 PoS 슬래싱 모델), AI 판정. 최종 신뢰 기반은 "성장 목적 유저 풀의 자기선택 + 낮은 상금의 어뷰저 진입장벽". **지금은 돈 로직 우선, 인증은 오라클 지갑 인터페이스만 열어두면 나중에 뭘 붙이든 컨트랙트 불변**

## 12. ⚡ Phase 1 개발 로드맵 — 돈 로직 (Sui Move 기준)

> 방침: 검증(인증)은 뒤로 미루고 **돈 로직부터**. 체크인 결과는 오라클 주소(임시로 팀이 수동 조작)가 제출하는 구조로 짜면, 나중에 BE 자동화/커뮤니티 투표/AI 뭘 붙여도 컨트랙트는 안 바뀜.
> **발표용 프로토(ver1~5 점진 구현)의 상세 스펙은 `정산_프로토타입/02_스펙/PROTO_SPEC.md`가 우선한다.** 아래 M0~M5는 프로토 이후의 본 개발 마일스톤.

### M0. 환경 세팅 (전원, 마일스톤 0)
- 전원 WSL에 **Sui CLI 설치** → `sui client` 동작 확인 → Devnet faucet으로 가짜 SUI 수령
- 레포에 `contracts/` 폴더 생성, `sui move new godsaeng` 으로 패키지 초기화 → 샘플 모듈 `sui move test` 통과 확인
- **전원이 로컬에서 테스트 초록불 뜨는 것 = M0 완료 기준.** 환경 문제는 여기서 다 털고 간다
- 레포 구조: `contracts/`(Move 패키지) `frontend/` `backend/` `docs/`

### M1. 챌린지 생성 + 참여 (예치)
- `create_challenge(체크인시간, 윈도우, 기간, 예치금, 최소인원)` → Challenge shared object 생성
- `join(challenge, Coin<SUI>)` — 참여자가 Coin 객체를 챌린지 vault에 예치
- 테스트: "방 만들고 3명 참여하면 vault 잔고 = 예치금×3인가"

### M2. 체크인 결과 수신 (오라클 인터페이스)
- `submit_results(challenge, day, 성공자목록)` — **오라클 주소만 호출 가능** (지금은 팀이 수동으로 호출, 나중에 BE가 대체)
- 시간 검증에 `Clock` 객체 사용 (UTC ms, KST 보정 주의)
- 테스트: "오라클 아닌 주소가 부르면 abort되는가", "같은 day 중복 제출 막히는가"

### M3. 정산 수식 (커브 + 스트리밍) ← 설계 완료, α만 회의 확정
- 볼록결합 환급 커브 `calc_refund(stake, d, total_days, alpha_bp)` + 스트리밍 drip을 **순수 함수로 분리**
- 순수 함수로 짜면 테스트가 "수식 검산"이 됨 — 시뮬레이터(`godsaeng_sim.py`)로 만든 기대값 표와 1:1 대조
- u64/u128 스케일링 (소수점 없음) 여기서 처음 체감하게 됨
- **핵심 테스트: 보존 법칙** — "∑수령 + dust = ∑예치, dust는 상계 이내"

### M4. claim (pull 패턴)
- 종료 후 각자 `claim(challenge)` 호출 → 원금+분배분의 Coin을 받아감
- 테스트: "이중 claim 막히는가", "탈락자는 환급분+적립 배당만 받는가"

### M5. 데모 모드
- 기간/윈도우가 생성 파라미터라면 자동 해결 — "1 day = 30초" 챌린지를 그냥 만들면 됨. M1에서 하드코딩만 안 하면 공짜

### 테스트 전략
- Move 유닛테스트(`sui move test`)가 기본. **시간 의존 로직은 test 시나리오에서 Clock을 조작/모킹**해서 "30일 시뮬레이션"을 몇 초 안에 돌림
- **풀시나리오 테스트 1개 = 데모 리허설**: 생성→3명 참여→매일 결과 제출(성공/실패 섞어서)→종료→claim→잔액 검증. 이거 통과하면 돈 로직은 믿어도 됨

### 협업 방식
- **M1은 셋이 모여서 몹 프로그래밍** (디스코드 화면공유) — 첫 조각에서 컨벤션이 다 정해짐
- M2부터 **"기능 구현 / 테스트 작성 / PR 리뷰" 롤을 마일스톤마다 교차** — 구현(.move)과 테스트 파일이 달라 conflict 없고, 테스트 작성자가 기능을 강제로 이해하게 됨 (AI 코드 검증 안전장치)
- Git: 각자 브랜치 → 마일스톤 하나 = PR 하나 → 리뷰 1명 이상 → main 머지. **main은 항상 테스트 전부 통과 상태 유지**

### 주차별 계획
- **1주차**: M0 + 전원 Sui Move 기초 (공식 docs.sui.io 튜토리얼, 크립토좀비 대체) + M1 몹 프로그래밍
- **2주차**: M2 + M3 (커브는 α 확정→기대값 표 재생성→코드 순서). 주말까지 풀시나리오 테스트 통과 목표
- **3주차**: M4 + M5 + **Testnet 배포** + 전원 실제 지갑으로 방 파고 참여 테스트 → 인터페이스 확정 → Phase 2(FE/BE) 착수
- **4주차**: FE/BE 통합 + 데모 리허설 + 발표자료
