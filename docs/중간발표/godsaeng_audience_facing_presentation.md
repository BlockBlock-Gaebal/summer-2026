# 갓생 내기 DeFi 프로토콜

> **의지력에 돈을 거는 Sui 기반 습관 챌린지**  
> 블록블록 여름 개발 프로젝트 중간발표

---

## 1. 기존 습관 내기 서비스의 문제

기존 습관 챌린지 서비스에서는 운영사가 다음 과정을 모두 통제합니다.

```text
사용자가 돈을 예치한다
        ↓
운영사가 예치금을 보관한다
        ↓
운영사가 성공·실패를 판정한다
        ↓
운영사가 환급액과 보상을 결정한다
```

> **사용자는 돈의 보관과 정산 규칙 모두를 운영사에 맡겨야 합니다.**

---

## 2. 우리의 해결 방식

```text
인증 결과는 Off-chain에서 제출
        ↓
예치금과 정산 규칙은 Smart Contract가 집행
        ↓
사용자가 계산된 보상을 직접 Claim
```

현실에서 습관을 실제로 수행했는지는 블록체인이 직접 확인할 수 없기 때문에 인증은 Off-chain에서 처리합니다.

반면 다음 항목은 Sui Move Contract에서 처리합니다.

- 참가자의 SUI 예치
- 환급액과 몰수액 계산
- 몰수금의 배당
- 최종 정산
- 사용자별 Claim

> **인증은 Off-chain, 돈과 정산 규칙은 On-chain으로 분리합니다.**

---

## 3. 현재까지 구현한 범위

현재는 전체 서비스 중 가장 중요한 **돈 정산 엔진**을 먼저 구현했습니다.

| 구현 항목 | 현재 상태 |
|---|:---:|
| Sui Move 정산 Contract | ✅ |
| 참가자 예치금 Vault | ✅ |
| 일별 결과 제출 | ✅ |
| 환급·몰수·배당 계산 | ✅ |
| 종료 후 개별 Claim | ✅ |
| Move Test | **21 / 21 통과** |
| Frontend · Backend | 다음 단계 |

### 현재 구현된 Contract 흐름

```text
create_challenge()
        ↓
join() — SUI 예치
        ↓
submit_results() — 일별 결과 제출
        ↓
finalize() — 최종 정산
        ↓
claim() — 각자 보상 수령
```

모든 정산 시나리오에서 다음 보존 법칙을 확인합니다.

> **참가자 전체 수령액 + Vault 잔액 = 최초 전체 예치금**

## 5. 차별점 ① 시간가중 환급

### 같은 탈락이라도 똑같이 처리하지 않습니다

```text
초반 탈락 → 적은 환급 → 큰 몰수
후반 탈락 → 많은 환급 → 작은 몰수
완주       → 원금 전액 환급
```

환급률은 다음 함수로 계산합니다.

$$
r(d)
=
\alpha\frac{d}{D}
+
(1-\alpha)\left(\frac{d}{D}\right)^2
$$

| 기호 | 의미 |
|---|---|
| $d$ | 참가자가 수행한 기간 |
| $D$ | 전체 챌린지 기간 |
| $r(d)$ | 탈락 시 돌려받는 원금의 비율 |
| $\alpha$ | 커브의 강도를 조절하는 값 |

기존 방식에서는 Day 2 탈락자와 Day 29 탈락자를 동일하게 처리할 수 있습니다.

우리 방식에서는 **얼마나 오랫동안 챌린지를 수행했는지**가 정산 결과에 반영됩니다.

---

## 6. 차별점 ② Streaming 분배

탈락자의 몰수금을 그날 바로 전부 나누지 않습니다.

### 즉시 분배 방식

```text
Day 2에 8 SUI 몰수
        ↓
Day 2 생존자에게 8 SUI 전부 지급
```

초반에는 일반적으로 다음 효과가 동시에 발생할 수 있습니다.

- 적응에 실패해 탈락자가 많이 발생
- 초반 탈락자는 환급률이 낮아 몰수액이 큼
- 발생한 몰수금을 당일에 전부 지급

따라서 보상이 초반에 집중되고, 후반에 계속 수행할 경제적 유인이 약해질 수 있습니다.

### Streaming 방식

```text
Day 2에 8 SUI 몰수
        ↓
Day 2부터 챌린지 종료일까지 나누어 방출
```

과거에 발생한 몰수금도 이후 계속 지급되기 때문에, **오래 생존할수록 누적해서 받을 수 있는 보상**이 유지됩니다.

## 7. 하나의 예시로 보는 정산 결과

### 조건

- A, B, C가 각각 10 SUI 예치
- 5일 동안 진행되는 챌린지
- B는 Day 2에 탈락
- C는 Day 4에 탈락
- A는 완주
- 아래 표는 선형 환급 기준인 $\alpha=1.0$ 예시

| 참가자 | 결과 | 원금 환급 | 누적 배당 | 최종 수령 |
|---|---:|---:|---:|---:|
| A | 완주 | 10.0 | +6.5 | **16.5 SUI** |
| C | Day 4 탈락 | 8.0 | +1.5 | **9.5 SUI** |
| B | Day 2 탈락 | 4.0 | 0 | **4.0 SUI** |
| **합계** |  |  |  | **30.0 SUI** |

이 결과는 다음 세 가지를 보여줍니다.

1. **B와 C는 모두 탈락했지만 수행 기간에 따라 수령액이 다릅니다.**
2. **C는 탈락 전에 이미 적립된 배당을 유지합니다.**
3. **최초 예치금 30 SUI가 정확히 보존됩니다.**

## 8. 왜 Sui 위에서 개발하는가?

| Sui의 특징 | 우리 서비스에서 얻는 이점 |
|---|---|
| **Object Model** | 챌린지 방을 독립된 On-chain Object로 관리 |
| **Move Resource Safety** | Coin을 임의로 복제하거나 소실하는 실수를 방지 |
| **Coin / Balance 구조** | 참가자의 예치금을 Vault에 명확하게 보관 |
| **PTB** | Coin 분리·예치·참가 등록을 하나의 Transaction으로 처리 |
| **Object 기반 실행** | 서로 다른 챌린지의 Transaction을 독립적으로 처리하기 쉬움 |
| **zkLogin** | 향후 일반 로그인과 유사한 사용자 경험 제공 |
| **Sponsored Transaction** | 향후 서비스가 사용자의 Gas를 대신 지불 가능 |

현재 프로토타입에는 다음 기능이 적용되어 있습니다.

- `Challenge` Shared Object
- `Coin<SUI>`를 이용한 참가
- `Balance<SUI>` 형태의 Vault
- Move의 Resource Safety
- 개별 Claim을 사용하는 Pull Pattern

PTB, zkLogin, Sponsored Transaction은 사용자 앱을 연결하는 단계에서 적용할 예정입니다.

> Sui를 선택한 이유는 단순히 속도가 빠르기 때문이 아니라,  
> **자산과 챌린지를 Object로 안전하게 다루는 구조가 서비스와 잘 맞기 때문입니다.**

## 9. 최종 발표까지 해야 할 일

### 1단계 — 정산 규칙 확정

- 성공으로 인정할 최소 달성률
- 한 번의 실패를 허용하는 Grace Day
- 환급 커브의 $\alpha$ 값
- 실패한 당일을 수행 기간에 포함할지
- 전원 탈락 시 남은 금액 처리
- 중도 참여가 가능한 마감 시점

### 2단계 — Move Contract 완성

- 확정된 정산 규칙 반영
- 기존 21개 Test 갱신
- 추가 Edge Case Test
- 정산 Event 추가
- Sui Testnet 배포

### 3단계 — 인증과 Backend Oracle

```text
사용자가 정해진 시간 안에 사진 제출
        ↓
방장 또는 다른 참가자가 승인
        ↓
Backend가 일별 결과 집계
        ↓
Oracle이 실패자 명단을 Contract에 제출
```

Backend는 **누가 성공하거나 실패했는지**만 제출합니다.

실제로 **얼마를 환급하고 몰수할지**는 Contract가 계산합니다.

### 4단계 — Frontend 연결

최종 발표까지 필요한 최소 화면은 다음과 같습니다.

- Wallet 연결
- 챌린지 목록
- 챌린지 생성
- 챌린지 참여와 SUI 예치
- 내 챌린지 Dashboard
- 사진 Check-in
- 종료 결과와 Claim

```text

        ↓
```
