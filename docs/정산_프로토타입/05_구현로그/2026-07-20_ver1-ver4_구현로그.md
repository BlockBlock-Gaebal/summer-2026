## 2. 버전별 상세

### ver1 — 기본형 (`8a294d7`)

스펙 §2의 구현 순서 3단계를 그대로 밟음. 각 단계 = 테스트 먼저 → 구현 → 초록불.

1. **create_challenge + join**: `vault = 30 SUI` 테스트 통과. Challenge를 shared object로 생성,
   `Coin<SUI>`를 `into_balance()`로 까서 vault에 합산. 이중 참여/금액 불일치 거부.
2. **submit_results**: 수동 day 카운터 (호출 1회 = day +1, Clock 미사용 — 스펙 §0).
   가드 4종: 오라클 전용 / 탈락자 재제출 방지 / 비참가자 명단 거부 / D회 초과 호출 거부.
3. **finalize + claim**: 스펙 기대값 **A=30 / B=0 / C=0, 보존 30=30** 통과.
   - finalize 가드 `current_day == total_days` (ver3의 "스트림 완납 전 종료 금지"를 미리 장착)
   - claim은 pull 패턴: finalize는 `claimable`에 숫자만 기록, 돈은 각자 찾아감.
     `claimable → 0 리셋 후 인출`이라 이중 claim은 `ENothingToClaim` abort
   - 엣지 2종: 전원 탈락 → 각자 원금 반환(0으로 나누기 회피, "불성립 시 돈은 원위치"),
     전원 성공 → 몰수 0에서 abort 없이 원금 반환

### ver2 — 금액반영형 (`880aa2e`)

- `create_challenge`에서 고정 `stake_amount` 제거, `join`이 임의 금액 수납 (0원만 거부)
- 분배: `몰수총액 × stake_i / 성공자지분합` — **u128 캐스팅이 여기서부터 필수**
  (40e9 × 20e9 = 8×10²⁰ > u64 상한 1.8×10¹⁹)
- 스펙 표 검산 통과: **A = 46,666,666,666 / D = 23,333,333,333 / dust 1 MIST** (합 70 SUI)
- 부수 확인: 균등 예치 시나리오(구 ver1 테스트)가 수정 없이 통과 → "ver1은 ver2의 특수 케이스" 실증

### ver3 ★ — 커브 + 스트리밍 (`51c53e0`)

스펙 지시 순서 T2 → T1 → T3 준수 (수식 검산 먼저, 시나리오는 나중).

- **calc_refund / calc_forfeit 순수 함수** (상태 접근 없음):
  `환급 = stake × [alpha_bp·d·D + (10000−alpha_bp)·d²] / (10000·D²)` — 통분해서 나눗셈 1회,
  분자 u128 누적 (u64 상한 근접). `⚠️ PLACEHOLDER` 주석 부착 (alpha_bp 회의 대기)
- **submit_results 3단계**: ① 탈락 기록 + `daily_drip += 몰수/(D−d+1)` (편입 시 1회 절단, §1-3 규약)
  → ② 생존자 지분합 → ③ drip을 생존자에게 지분 비례 적립.
  ①이 ③보다 먼저라서 "그날 탈락자는 그날 배당 제외"가 순서만으로 성립
- **전멸 임시 규칙** (스펙 §4 ⚠️): 생존자 0 감지 → 조기 ENDED, 탈락자 커브 환급만 확정,
  미방출 스트림은 vault 잔류 (보존 테스트에서 별도 항으로 검증)
- 테스트 3종 전부 스펙 표 숫자 그대로 통과:
  - **T2** (α=0.3, D=5): 환급 0.88 / 2.32 / 4.32 / 6.88 / 10.00 SUI — 5점 전부 MIST 정수 일치
  - **T1** (α=1.0, dust 0): **A=16.5 / C=9.5 / B=4.0 SUI** + 중간 검증(C가 day4 탈락 시점에
    배당 1.5 SUI 기보유 — 킬러 슬라이드 수치)
  - **T3** (dust): A=C=11,999,999,998 / dust 4 ≤ 상계 20 (※ ver4에서 갱신됨, §4-1 참고)

### ver4 — 중도 참여 (`c9ce509`)

- **배당 회계를 acc_per_share 패턴으로 전환** (Aave/MasterChef 정석, 스펙 §5가 지목한 방식):
  - 매일: `acc += daily_drip × SCALE(1e12) / 생존자지분합` — 개인별 순회 적립 대신 O(1)
  - 개인 배당: `stake × (acc_정산시점 − acc_참여시점) / SCALE` — 탈락일 또는 finalize에 일괄 정산
  - join 시 acc 스냅샷(`acc_entry`) → **참여 이전 배당 제외가 뺄셈으로 자동 보장** (로직 아닌 수학)
- **join 개방**: `status == ACTIVE && current_day < total_days`면 참여 가능. `start_day = current_day + 1`
- **중도참여자 커브**: 개인 타임라인 `d_개인 = 탈락일−start_day+1, D_개인 = total_days−start_day+1`
  (⚠️ 임시 규칙 — §4-3 참고)
- 테스트 2종 (기대값 손검산, 아래 §4-2에 검산 근거):
  - **V4-1** 배당 제외: A=13.75 / C=12.25 / B=4.0 (C가 참여 전 day2 배당 1.5를 못 받음, dust 0)
  - **V4-2** 개인 커브: A=18,583,333,334 / C=7,416,666,666 / B=4.0 (합 정확히 30 SUI, dust 0)

## 4. ⚠️ 스펙과 다르게 판단한 지점 (정합성 체크 요청 목록)

> 검토자용 핵심 섹션. 각 항목: 무엇을 / 왜 / 어디서 확인.

### 4-1. T3 기대값 1 MIST 이동 (ver4 커밋에서)

- **변경**: T3 기대 A=C `11,999,999,998 → 11,999,999,999`, dust `4 → 2`
- **원인**: acc_per_share 전환으로 배당 절단(floor)이 "매일 1회" → "정산 시점 1회"로 이동.
  ver3 방식: `floor(666,666,666.5) × 3일 = 1,999,999,998` / ver4 방식: `floor(1,999,999,999.5) = 1,999,999,999`
- **판단 근거**: 절단 횟수 감소 = 유저에게 유리한 방향, dust 상계(≤ D×(인원+1))는 그대로 성립.
  스펙 §1-3의 "drip 편입 시 1회 절단" 규약은 불변 (그건 drip 쪽 얘기). **스펙 T3 표의 숫자와
  달라진 것은 사실이므로, 회의에서 "배당 절단 규약: 정산 시 1회"로 명문화 필요**
- **확인**: `tests/challenge_tests.move`의 `test_t3_dust_bound` 주석에 양쪽 계산 병기

### 4-2. V4 테스트 시나리오를 스펙 스케치와 다르게 구성

- **스펙 §5 스케치**: "A, B day1 참여, C day3 참여, **B day4 탈락** → C가 day2 이전 배당을 못 받는지 검증"
- **문제**: B가 day4에 탈락하면 C 참여(day3) 이전에 배당이 아예 발생하지 않아 검증이 공허함
- **조정**: B 탈락을 **day2로 이동** → C 참여 전 배당 1.5 SUI가 실재 → 제외 검증이 유효해짐.
  검증 대상 자체("참여 이전 배당 제외")는 스펙 그대로
- **V4-1 검산**: day2 B 몰수 6 → drip 1.5, 생존자 A 혼자 +1.5 / day3~5 A·C 각 +0.75
  → A = 10+1.5+2.25 = 13.75, C = 10+2.25 = 12.25, B = r(2)=4.0. 합 30, dust 0
- **V4-2 검산** (V4-1에서 C가 day4 탈락): C 개인 타임라인 d=2, D=3 → 환급 10×2/3 = 6,666,666,666.
  몰수 3,333,333,334 → drip += /2 = 1,666,666,667 → day4~5 A 독식.
  A = 10 + 1.5 + 0.75 + 3.166,666,667 + 3.166,666,667 = 18,583,333,334 / C = 6,666,666,666 + 0.75e9
  = 7,416,666,666 / B = 4.0. **합이 정확히 30 SUI (dust 0)** — 전 나눗셈이 떨어지게 설계

### 4-3. 중도참여자 커브의 임시 규칙 (미결 안건 선반영)

- 스펙 §5가 "중도참여자의 D(기간) 정의"를 **튜닝 미결**로 남김 → 임시 규칙으로
  `d_개인 = failed_day − start_day + 1`, `D_개인 = total_days − start_day + 1` 채택
- 근거: start_day=1이면 ver3 수식과 정확히 일치하는 최소 일반화. 몰수분의 스트림 분모는
  전역 잔여일 `(total_days − d + 1)` 유지 (모든 스트림 종점 = D 불변식 보존)
- 위치: `sources/challenge.move`의 `personal_refund` (⚠️ 주석)

### 4-4. 전멸 규칙의 해석 보강

- 스펙 임시 규칙("조기 ENDED + 미방출 스트림 vault 잔류")에는 **탈락자 환급 처리가 명시 안 됨**.
  finalize를 안 거치므로 환급을 어디서 확정하는지가 공백 → `submit_results`의 전멸 분기에서
  커브 환급을 claimable에 즉시 확정하도록 구현 (안 그러면 전멸 시 환급이 영구 잠김)
- 그날 탈락자의 기적립 배당은 ① 단계에서 이미 정산되므로 이중 지급 없음
- `submit_results`에 `status == ACTIVE` 가드 추가 (전멸 조기종료 후 재호출 차단) — 스펙에 없던 가드

### 4-5. 기타 구현 판단 (소소)

| 판단 | 내용 / 근거 |
|---|---|
| `claim`이 `Coin<SUI>` 반환 | sender에게 transfer하는 대신 반환 — PTB 합성 가능, 테스트에서 금액 assert 용이 |
| `finalize` 권한 제한 없음 | 결과가 결정적(상태만으로 계산)이라 누구나 호출 가능해도 무해 |
| 파일명 변경 | `godsaeng.move → challenge.move` (모듈명 `godsaeng::challenge`와 일치) |
| 에러코드 교체 | ver2에서 `EWrongStakeAmount(0) → EZeroStake(0)` — flat 균등 베팅 소멸에 따름 |
| join 마감 임시값 | `current_day < total_days` (마지막 day 시작 후 불가) — ⚠️ 참여 마감선 미결, 파라미터화 대기 |
| ver3 커밋에서 구 풀시나리오 테스트 제거 | flat 정산 의미 소멸 → T1/T3로 대체. 구 테스트는 `8a294d7`/`880aa2e`에 보존 |

## 6. Move 개념 정리 (팀 룰 4번 — 팀원 설명용 치트시트)

| 개념 | 한 줄 설명 | 코드 위치 |
|---|---|---|
| shared object | 컨트랙트 전역 상태 대신 "객체" 단위. `Challenge` 하나 = 방 하나, `transfer::share_object`로 누구나 참조 가능하게 공개 | `create_challenge` |
| `Coin` vs `Balance` | 돈이 msg.value로 딸려오는 게 아니라 `Coin<SUI>` 객체를 인자로 넘김. 내부 보관용은 `Balance` (UID 없는 잔액 타입) | `join`, `Challenge.vault` |
| 리소스 안전성 | `Coin`은 drop 불가 → 받았으면 반드시 어딘가로 보내야 컴파일됨. "이체 깜빡" 버그가 언어 차원 차단 | `join`, `claim` |
| `Table` 순회 불가 | Solidity mapping처럼 키 순회 불가 → `participant_list` vector 병행 유지 | 스펙 §1-1 ★ |
| `do!` / `do_ref!` | vector 매크로 순회 (for 루프 대체, 인라인 전개 람다) | `submit_results`, `finalize` |
| modifier 없음 | 권한 검사는 함수 첫 줄 `assert!(ctx.sender() == ch.oracle, ENotOracle)` | `submit_results` |
| abort 코드 | `revert CustomError()` 대신 u64 상수 + `expected_failure(abort_code=...)`로 테스트 매칭 | 에러 상수 블록 |
| u128 캐스팅 | 곱셈 오버플로 시 자동 abort → 큰 곱셈은 `(x as u128)` 승격 후 마지막에 u64로 | `calc_refund`, `pending_dividend` |
| `test_scenario` | 멀티유저 트랜잭션 시뮬레이션: `next_tx(주소)` → `take_shared` → `return_shared` | 테스트 헬퍼 전부 |
| acc_per_share | "지분 1단위가 여태 받았어야 할 배당 누적치"만 O(1) 갱신, 개인 몫은 스냅샷 뺄셈. 비유: 출근 시각 찍어두면 그 이전 시급은 자동 제외 | `acc_per_share`, `acc_entry` |
| pull 패턴 | 일괄 송금 루프 금지 — 한 명의 실패가 전체 정산을 막는 안티패턴 회피. 각자 `claim` | `finalize`/`claim` 분리 |
