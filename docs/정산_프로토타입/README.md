## 읽는 순서 (처음 보는 사람 기준)

1. **`01_설계문서/01_CURVE_DESIGN_ver1-3.md`** — 왜 즉시분배가 아니라 스트리밍인지, 왜 선형이 아니라 볼록 커브인지. 명제 1~4 + 증명. 여기가 전체 설계의 기초 공리를 세우는 곳.
2. **`01_설계문서/02_VER4_DESIGN.md`** — 중도 참여를 열면 생기는 세 가지 문제(회계/커브시계/타이밍게임)와 세 가지 설계안(개방형/빈티지분리/NAV진입가) 비교. 명제 5~8. 1번 문서를 읽었다는 전제로 씀.
3. **`02_스펙/PROTO_SPEC.md`** — 위 설계를 실제 Move 코드로 어떻게 짤지의 구체적 스펙. ver1~5 각각의 상태 모델, 함수 시그니처, **기대값이 하드코딩된 테스트 시나리오**. Claude Code에 이 문서를 던지면 바로 구현 시작 가능.
4. **`03_시뮬레이터/`** — 위 두 설계문서의 모든 수치·그래프가 나온 소스. 눈으로 검산하고 싶거나 파라미터(α, 몬테카를로 hazard 등)를 바꿔서 새 시나리오를 뽑고 싶을 때 여기를 돌림.

## 핵심 설계 요약 (한 화면 정리)

**정산 방식**: 몰수금을 즉시 나눠주지 않고 잔여 기간에 걸쳐 균등 방출하는 **스트리밍(베스팅) 분배**.

**몰수 커브**: `r(d) = α·(d/D) + (1-α)·(d/D)²` — 선형과 이차의 볼록결합. α는 "얼마나 볼록한가"를 조절하는 유일한 파라미터. 권장 초기값 **α=0.3**, 탐색 구간 [0.2, 0.4].

**중도 참여 (ver4, 스트레치 골)**: 커브는 개인 잔여기간으로 정규화(강제됨 — 안 그러면 완주해도 손해 보는 "기부 함정" 발생). 늦게 들어올수록 유리해지는 "찍먹 차익" 문제가 있고, MVP는 **개방형 + 참여 마감선**으로 방어, 공개방 정공법은 **빈티지 분리**(로드맵).

**한 줄 요약**: "커브 방향을 고른 게 아니라, 인센티브 조건(생존 임금이 노력 비용을 이겨야 한다)을 정의하고 그 조건을 만족하는 함수족을 유도했다."
